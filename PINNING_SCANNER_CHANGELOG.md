# 박제 스캐너 (woos_pinning_scanner) 변경 이력

A형(박제형) 발사 종목을 실시간 탐지하는 스캐너. 빗썸 전 종목 호가를 폴링해
세력 매집(가두리 박제)을 포착하고, 가두리 풀림 시점에 발사 임박 알림을 보낸다.

---

## 검증 근거 (R2 과거 데이터 백테스트)

박제형 = 발사 직전 두 조건을 **동시에** 만족하는 종목:

1. **매수벽 억눌림** — 매수벽비율(총매수잔량/총매도잔량)이 1.0 미만인 시간이
   직전 2시간 중 70% 이상. 세력이 매도벽으로 가격 위를 막고 흡수.
2. **거래 죽음** — 분당 체결 1건 미만. 매집 완료 후 인위적 정적.

| 단계 | 내용 | 백테스트 |
|------|------|----------|
| STAGE 1 (박제 포착) | 억눌림 70%+ AND 거래죽음 분당<1 | 6/6 (ICX·VVV·STRAX·ORCA 포착, ZKP·WAVES 제외) |
| STAGE 2 (발사 임박) | 박제 후보의 매수벽 1.0 돌파 + 매도벽 50%+ 급감 | 4/4 (ICX·VVV·STRAX·ORCA 풀림 검출) |

**핵심 원칙**
- 거래 죽음은 A형의 필요조건이나 충분조건은 아님 (거래 죽은 ATOM·NEAR는 비발사).
  억눌림과 결합해야 변별됨.
- 봇 매집 누적액 상위로 후보를 조이는 것은 역효과 (대형코인 ONDO·XLM이 상위 점유,
  박제형은 거래 죽여 누적액 바닥). 박제 서명 자체가 1차 필터여야 함.
- 장악률(개입/빗썸유통량)은 발사 전 변별력 없음이 확인됨 → 알림에 **참고용**으로만 표시.
- 가두리 풀림은 전 종목에선 상시 노이즈. **박제 후보에 게이팅했을 때만** 유효.
- A형 전용. B형(거래량 변곡: ZKP·WAVES·KERNEL·WLD)은 이 서명으로 안 잡힘 — 별도 과제.

## 버전별 변경

### v4 (현재)
- 한글 티커 적용 — `load_korean_names()`로 빗썸 API에서 종목명 463종 로딩,
  `종목: 한글명(TICKER)` 표시.
- 알림 형식을 기존 FSM `build_signal` 형식으로 통일 (구분선·매수가·손절·TP).
- STAGE 2에 추가 정보:
  - 순매수 방향 — 🟢적극매집 / 🔴흡수매집 (봇 매수-매도 부호).
  - 장악률 — "참고" 라벨 (발사 근거 아님). 박제 후보에만 계산해 경량.
  - 위쪽 매물대 — 자리 확보, 데이터 있을 때만 표시 (현재 미계산).
- 손절 -8% / TP1(+5%)→50% / TP2(+10%)→30% / TP3(+15%)→20% / 보유한계 4H.
- **HTML 이스케이프 수정** — 메시지 내 `<` 기호로 인한 텔레그램 400 Bad Request 해결.
  `매수벽<1.0` → `매수벽 1.0미만`, `<b>` 태그만 보존하고 나머지 `< > &` 이스케이프.

### v3
- 텔레그램 알림을 기존 알파봇 `.env`(`TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID_GROUP`)
  방식으로 통일. `TG_TOKEN` 직접 참조 제거 (NameError 크래시 해결).
- `.env` 경로를 `woos-alpha-bot/.env`로 고정.

### v2
- 거래 죽음 측정을 VM 로컬 틱 파일(`~/woos_logs/tick_bithumb_*.csv.gz`) 직접 읽기로 변경
  (빗썸 trades API 근사보다 정확).
- STAGE 2 (발사 임박 트리거) 추가, 박제 후보에만 게이팅.

### v1
- STAGE 1 (박제 포착) 최초 구현. 빗썸 v1/orderbook 복수 종목 API로
  매수벽비율 폴링, 2시간 슬라이딩 윈도우.

## 운영

- 실행: `systemd` 서비스 `woos-pinning.service` (10초 폴링, 상시 가동).
- 데이터: 빗썸 `v1/orderbook` (호가, 복수 종목) + VM 로컬 틱 (거래 죽음·장악률).
- 알림: 박제 포착 📌 / 발사 임박 🚀, 기존 알파봇 텔레그램 방.
- 주의: 슬라이딩 윈도우 2시간이라 기동 후 약 1.5시간 지나야 첫 박제 판정 가능.
- 표본이 작아(10종) 임계값(억눌림 70%, 거래죽음 분당<1)은 실전 후보 분포 보며 조정 필요.
